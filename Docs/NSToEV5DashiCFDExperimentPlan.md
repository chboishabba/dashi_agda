# NS-to-EV5 dashiCFD Experiment Plan

## Purpose

Record a numerical experiment plan for testing the Navier-Stokes-to-EV5
projection against concrete 2D vorticity / LES traces generated by the sibling
`../dashiCFD` repository.

This is an evidence-gathering experiment only. It is not a proof of the
NS-to-EV5 projection, not a proof of 3D Navier-Stokes regularity, and not a
Clay promotion. A 2D CFD/LES diagnostic can falsify or guide a projection
hypothesis, but it cannot prove the 3D Navier-Stokes Millennium problem.

## Available dashiCFD Surface

The sibling `../dashiCFD` repo already exposes scripts that can generate 2D
vorticity and LES evidence surfaces:

- `scripts/make_truth.py`: writes truth trajectory NPZ/JSON artifacts with
  `omega_snapshots`, `steps`, `energy`, and `enstrophy`.
- `run_les_gpu.py`: GPU LES runs that emit `step,enstrophy` CSV logs.
- `run_v4_snapshots.py`: canonical end-to-end CFD pipeline for LES -> encode
  -> learn -> rollout -> decode snapshots and metrics.
- `dashi_les_vorticity_codec.py` and `dashi_les_vorticity_codec_v2.py`:
  2D vorticity simulations with DASHI structural codec outputs and enstrophy
  curves.
- `dashi_cfd_operator_v3.py` and `dashi_cfd_operator_v4.py`: baseline LES
  rollouts, proxy/decode comparisons, energy/enstrophy drift, and plotted
  enstrophy traces.
- `scripts/plot_enstrophy.py`: plotting path for JSON/CSV enstrophy logs.

These traces are sufficient to build a reproducible numerical probe for the
candidate NS-to-EV5 projection.

## Concrete 2D Turbulence Experiment

Run decaying and resonantly forced 2D vorticity/LES cases on a periodic square.
Use `scripts/make_truth.py` as the preferred source when shell diagnostics need
full vorticity snapshots, and use `run_les_gpu.py` when only total enstrophy CSV
is needed for backend stability checks.

For each stored vorticity snapshot `omega(x,y,t)`, compute the Fourier
coefficients `omega_hat(k,t)`. For dyadic shell
`S_j = { k : 2^j <= |k| < 2^(j+1) }`, record shell enstrophy
`E_j(t) = 0.5 * sum_{k in S_j} |omega_hat(k,t)|^2`, using one declared FFT
normalization. Also record total shell enstrophy
`E_shell(t) = sum_j E_j(t)` and compare it with the run's existing total
enstrophy diagnostic as a normalization guard.

The first-pass matrix should include:

- decaying runs: at least two seeds and two grids, for example `N=128` and
  `N=256`, with fixed `dt`, `nu0`, `Cs`, stride, dtype, and backend recorded;
- forced runs: a low-shell periodic forcing and a high-shell periodic forcing,
  each with forcing amplitude, frequency, active shell, phase, and schedule
  recorded in the manifest;
- a no-forcing control with the same numerical settings as each forced run;
- CPU/GPU comparison when both backends are available.

If the sibling repo does not yet expose a forcing flag, the forced-run adapter
is a new experiment adapter and must not be described as already present in
`../dashiCFD`.

## Proposed Extraction

For each selected run, export a structured trace with:

- run metadata: script, commit/hash if available, seed, grid size, timestep,
  viscosity/LES settings, backend, dtype, and output artifact paths;
- dyadic shell depth `j` from the vorticity/velocity spectrum;
- per-shell enstrophy weights `E_j(t)`;
- total enstrophy `Z(t)` and optional energy `E_flow(t)`;
- inter-shell flux proxy or adjacent-shell transfer estimate when available;
- codec/proxy state summaries that can be aligned with the shell data.

The first-pass shelling should use dyadic Fourier annuli. For a shell
`2^j <= |k| < 2^(j+1)`, record the enstrophy contribution as the shell weight.
Normalize each time slice in a declared way: raw weights, sum-normalized
weights, or log-compressed weights. The normalization must be part of the
artifact manifest so the EV5 candidate is reproducible.

## EV5 Encoding

Encode each time slice into an EV5 candidate over the five FRACTRAN lanes
`2,3,5,7,11` by first extracting the top enstrophy-bearing shells and then
recording the current candidate lane semantics:

- lane 2: active dyadic shell depth `j_max`, the shell with largest nonzero
  `E_j(t)`;
- lane 3: adjacent cascade ratio `E_{j_max+1}(t) / E_{j_max}(t)`;
- lane 5: secondary top-shell occupancy, excluding the peak shell;
- lane 7: peak-shell enstrophy fraction, the termination-critical lane;
- lane 11: `Z/3` phase/coherence proxy outside the canonical FRACTRAN rules.

The EV5 state for a time slice is a manifest-declared integer quantization of
these five lane buckets. The encoding must preserve enough metadata to
reconstruct the peak shell, the adjacent ratio, the secondary-shell mass, the
peak enstrophy fraction, and the phase/coherence proxy.

This lane assignment is a candidate, not a theorem. The experiment should
include alternative bucketizations and record whether the monotonicity and
nonresonance checks are stable under those alternatives.

## Checks

For each encoded EV5 trace:

- Q monotonicity: define a scalar `Q(t)` in the manifest, initially the
  FractranCOL `E` value of the encoded EV5 state plus an optional residual-tail
  penalty. In decaying runs, `Q(t)` must be nonincreasing after a declared
  burn-in window except for a stated numerical tolerance and a bounded number of
  one-step violations.
- r17 behavior: check whether tail activity in lane `7` steps down in the
  expected `r17` direction when the trace enters a dissipative tail regime. This
  is a diagnostic comparison only.
- Resonant forcing cycle test: in forced runs, flag repeated EV5 states or
  short cycles whose period locks to the forcing period while shell enstrophy
  continues to circulate between shells. Record the cycle length, forcing
  period, shell indices involved, and whether the same cycle appears in the
  no-forcing control.
- Shell contradiction check: flag cases where `E_j(t)` transfers upward or
  cycles while the encoded `Q(t)` claims descent.
- Stability: rerun across seeds, grid sizes, timesteps, and CPU/GPU backends
  where feasible.
- Falsification rule: preserve failures. The candidate encoding is falsified
  for this experiment family if any reproducible run has a tolerated-shell
  normalization guard pass but shows either persistent `Q(t)` increase in a
  decaying run, a resonant EV5 cycle that survives the no-forcing comparison, or
  shell-enstrophy movement that contradicts the encoded descent rule.

## Receipt Boundary

The desired receipt should certify only the following:

- `../dashiCFD` can provide numerical 2D vorticity/LES enstrophy traces;
- the experiment extracts dyadic shell depth and shell enstrophy weights
  `E_j(t)`;
- the derived shell-enstrophy features are encoded into an EV5 candidate;
- the candidate is checked against Q monotonicity, FractranCOL `r17`
  diagnostics, and resonant-forcing cycle behavior;
- falsifying outcomes are retained as negative evidence;
- all outcomes are evidence only, with no proof or Clay promotion.

  `step,time,lane2,lane3,lane5,lane7,lane11,energy`;

The Agda receipt, if present, should remain a governance receipt for the plan
and not claim that the numerical evidence proves the NS-to-EV5 projection.
