# Autonomous Proxy Generalization

This theorem surface formalises the scientific boundary required by the current DASHI CFD experiments.

## CORE and CFD lanes

The **CORE** lane evolves a compact proxy state under an operator:

```text
z(t + 1) = F(z(t))
```

The autonomous rollout receives only:

- the learned operator,
- an initial proxy state,
- and a time index.

It does **not** receive the future LES/CFD trajectory. Consequently, the reference tail cannot affect the rollout by construction.

The **CFD/LES** lane has two restricted roles:

1. provide a finite training trace from which an operator may be learned;
2. provide an optional reference timeline for holdout comparison.

Decoding is an observer chart from proxy state to a rendered field. Changing the decode chart does not alter the CORE timeline.

## Finite-prefix training

`FiniteTrace State` uses `Fin length` indices. A learner therefore receives a finite trace object rather than an unbounded reference timeline:

```agda
Learner State = FiniteTrace State → Operator State
```

The resulting timeline is generated by repeated application of the learned operator:

```agda
autonomousFromTraining learn training seed t
```

## Generalization boundary

Evaluation relations are indexed:

- `inSampleReplay`
- `heldOutTime`
- `unseenInitialCondition`
- `shiftedParameters`
- `extendedHorizon`

`GeneralizationWitness` has constructors for the four genuine holdout relations and deliberately has no constructor for `inSampleReplay`.

The theorem

```agda
inSampleReplayIsNotGeneralization :
  GeneralizationWitness inSampleReplay → ⊥
```

therefore prevents a same-trajectory replay score from being promoted as generalization evidence.

## Holdout comparison

`HoldoutComparison` combines:

- an autonomous learned operator,
- a seed,
- a decode map,
- a CFD/LES reference timeline,
- a comparison function,
- and a typed generalization witness.

Its `predicted` timeline is generated without reference input. The reference is used only by `measured` after prediction.

## Backend parity

Vulkan, CPU, or another backend is an implementation of the same operator semantics only when accompanied by a `BackendParity` witness:

```agda
stepParity : (x : State) → runStep x ≡ specification.step x
```

This captures the repository's current requirement that GPU activation and performance claims must not silently substitute a different operator.

## Current scope

This module proves structural separation and promotion boundaries. It does not claim that the present linear proxy operator generalises on Navier–Stokes trajectories. Empirical promotion still requires:

- held-out time windows,
- unseen initial conditions or seeds,
- extended autonomous horizons,
- parameter-shift tests,
- and physical/quotient diagnostics measured only after rollout.
